1. Introduction
1.1 Purpose of this Document
This Software Design Specification (SDS) defines the complete architectural, structural, and behavioural design of the C++ Packet Filtering Firewall. It translates every requirement stated in the SRS (v1.0) into precise design decisions, class hierarchies, data structure assignments, algorithms, and interface contracts that a developer can follow directly to write code.
It is written to satisfy: (a) the DSA project viva/submission (demonstrating all 5 data structures), (b) the SE subject deliverable (Pressman design models, UML, CLO-3), (c) the DBMS subject (flat-file persistence with SQL-equivalent querying), and (d) the InfoSec subject (threat detection, CIA Triad implementation).
1.2 Scope
The system is a command-line C++ application that:
Accepts simulated network packets as input (C++ structs, not live pcap).
Evaluates each packet against a rule set using 5 data structures (LinkedList, BST, HashMap, Stack, Queue).
Applies BLOCK or ALLOW decisions and logs every event to a CSV file.
Detects anomalous traffic patterns (SYN flood, port scan).
Exposes a full CLI for administrator management of rules, blacklists, and statistics.
GUI is defined as a future increment and is architecturally planned (MVC) but not implemented in this version.

2. System Architecture
2.1 Architectural Style
The firewall uses a hybrid architecture combining three styles:
Pipes and Filters — packets flow through a linear pipeline of processing stages (Capture → Parse → Blacklist Check → Rule Match → Action → Log). Each stage is a self-contained filter.
Layered Architecture — four tiers: Presentation (CLI), Application Logic (RuleEngine, TrafficMonitor), Data Access (FileLogger, ConfigManager), and Data (rules.conf, firewall.log).
Repository Architecture — a shared in-memory repository (FirewallState) stores the active rule set, blacklist, whitelist, statistics, and queue. All modules read from and write to this shared state.
Rationale: Pressman Ch10 defines these as standard architectural patterns. The Pipes-and-Filters pattern directly maps to how real network firewalls (iptables, FortiGate) process packets. Layered architecture ensures the future GUI can replace the CLI without touching the engine.
2.2 Module Overview
Six top-level modules
1. PacketSimulator   — Generates test packets (Packet structs). Not part of production core.
2. PacketProcessor   — Main pipeline: receives packets, coordinates all inspection stages.
3. RuleEngine        — Owns the rule data structures (LinkedList, BST, HashMap).
4. TrafficMonitor    — Detects anomalies: SYN flood, port scan.
5. FileLogger        — Writes CSV log entries and reads them back for queries.
6. AdminCLI          — Parses and dispatches all administrator commands.

3.5 Class: BST (Rule lookup — priority-keyed)
Binary Search Tree keyed on Rule::priority. Provides O(log n) lookup of the highest-priority matching rule. This is the critical performance path for every packet decision.
Attributes:
RuleNode* root — pointer to BST root.
Methods:
void insert(Rule r) — inserts by priority key. If priority already exists, uses ruleID as tie-breaker.
Rule* findBestMatch(const Packet& p) — in-order traversal from lowest priority number (highest priority), returns first Rule that matches the packet. O(log n) in a balanced tree.
bool remove(int ruleID) — finds and removes node, rebalancing with in-order successor. O(log n).
void inOrder(vector<Rule>& out) — fills vector with rules in priority order (for listing).
int height() — returns tree height (useful for performance analysis in DSA viva).
bool isBalanced() — checks if tree is balanced (AVL property, for analysis).
Design note: In the worst case (rules inserted in sorted priority order), the BST degenerates to O(n). For the viva, discuss this trade-off and mention AVL trees as the self-balancing extension.
3.6 Class: HashMap (IP blacklist, whitelist, SYN counters)
Hash table mapping string keys (IP addresses) to boolean (blacklist/whitelist) or integer (SYN counter). O(1) average lookup.
Design decisions:
Hash function: djb2 (Dan Bernstein) — simple, fast, good distribution for IP strings.
Collision resolution: Separate chaining (each bucket is a linked list of entries).
Load factor threshold: 0.75. When exceeded, rehash to 2x bucket count.
Bucket count: Start with 1024 (power of 2 for fast modulo with bitwise AND).
Attributes:
vector<list<pair<string,int>>> buckets — bucket array of chained pairs.
int count — current number of entries.
int bucketCount — current number of buckets.
Methods:
void insert(const string& key, int value=1) — hashes key, inserts or updates in bucket chain.
int get(const string& key) — returns value for key, -1 if not found.
bool contains(const string& key) — returns true if key present.
void remove(const string& key) — removes entry from chain.
void increment(const string& key) — increments integer value for key (used for SYN counters).
vector<pair<string,int>> topN(int n) — returns N entries with highest values (for top blocked IPs).
float loadFactor() — returns count / bucketCount.
void rehash() — doubles bucket count and re-inserts all entries.
int hash(const string& key) — private: djb2 hash function.

4. Data Structure Design (Complete Reference)
This section is the primary DSA documentation. Each data structure is fully specified with: role in the firewall, C++ implementation notes, complexity analysis, and DSA viva talking points.
4.1 LinkedList — Rule Storage and Ordered Traversal
LinkedList Quick Reference
Purpose: Store all rules in order. Used for: listing rules, saving to file, loading from file.
Node type: RuleNode (contains Rule, next*, left*, right* — shared with BST)
Key operation: insertSorted() maintains priority order during bulk load.
Complexity: Insert O(1) tail / O(n) sorted | Remove O(n) | Find O(n) | Traverse O(n)
C++ pseudocode — Node definition:
struct RuleNode {
    Rule rule;
    RuleNode* next  = nullptr;   // LinkedList link
    RuleNode* left  = nullptr;   // BST left child
    RuleNode* right = nullptr;   // BST right child
};
C++ pseudocode — insert():
void LinkedList::insert(Rule r) {
    RuleNode* node = new RuleNode{r};
    if (!head) { head = tail = node; }
    else        { tail->next = node; tail = node; }
    size++;
}
C++ pseudocode — remove():
bool LinkedList::remove(int ruleID) {
    RuleNode* prev = nullptr;
    RuleNode* curr = head;
    while (curr) {
        if (curr->rule.ruleID == ruleID) {
            if (prev) prev->next = curr->next;
            else      head = curr->next;
            if (curr == tail) tail = prev;
            delete curr; size--; return true;
        }
        prev = curr; curr = curr->next;
    }
    return false;  // not found
}
4.2 BST — Priority-Keyed Rule Lookup
BST Quick Reference
Purpose: O(log n) lookup of the highest-priority matching rule for each packet.
Key: Rule::priority (integer, lower = higher priority).
Traversal: In-order traversal visits rules in priority order (lowest number first).
Complexity: Insert O(log n) avg | Search O(log n) avg | Remove O(log n) avg | Worst O(n) sorted input
Viva point: Discuss degenerate case (rules inserted in sorted priority order → linked list).
C++ pseudocode — insert():
void BST::insert(RuleNode* &node, Rule r) {
    if (!node) { node = new RuleNode{r}; return; }
    if (r.priority < node->rule.priority)
        insert(node->left, r);
    else if (r.priority > node->rule.priority)
        insert(node->right, r);
    else // same priority: use ruleID as tie-breaker
        if (r.ruleID < node->rule.ruleID)
            insert(node->left, r);
        else
            insert(node->right, r);
}
C++ pseudocode — findBestMatch() (in-order, returns first match):
Rule* BST::findBestMatch(RuleNode* node, const Packet& p) {
    if (!node) return nullptr;
    // Check left subtree first (higher priority = lower number)
    Rule* found = findBestMatch(node->left, p);
    if (found) return found;
    // Check this node
    if (node->rule.matches(p)) return &node->rule;
    // Check right subtree (lower priority)
    return findBestMatch(node->right, p);
}
C++ pseudocode — remove() (in-order successor method):
RuleNode* BST::removeNode(RuleNode* node, int ruleID) {
    if (!node) return nullptr;
    if (ruleID < node->rule.ruleID)
        node->left  = removeNode(node->left,  ruleID);
    else if (ruleID > node->rule.ruleID)
        node->right = removeNode(node->right, ruleID);
    else {
        if (!node->left)  { RuleNode* r = node->right; delete node; return r; }
        if (!node->right) { RuleNode* l = node->left;  delete node; return l; }
        // Two children: replace with in-order successor
        RuleNode* succ = minNode(node->right);
        node->rule = succ->rule;
        node->right = removeNode(node->right, succ->rule.ruleID);
    }
    return node;
}
4.3 HashMap — IP Blacklist / Whitelist / SYN Counters
HashMap Quick Reference
Purpose: O(1) average lookup for IP-keyed data (blacklist, whitelist, SYN flood counters).
Hash function: djb2 — hash = 5381; for each char: hash = hash*33 ^ char;
Collision resolution: Separate chaining (each bucket = linked list of pairs).
Load factor threshold: 0.75 — triggers rehash to 2x bucket count.
Initial bucket count: 1024 (power of 2 for fast modulo).
Complexity: Insert O(1) avg | Get O(1) avg | Remove O(1) avg | Worst O(n) all same bucket
C++ pseudocode — hash function:
int HashMap::hash(const string& key) {
    unsigned long h = 5381;
    for (char c : key)
        h = ((h << 5) + h) ^ (unsigned char)c;  // h*33 ^ c
    return (int)(h % bucketCount);
}
C++ pseudocode — insert():
void HashMap::insert(const string& key, int value) {
    if (loadFactor() > 0.75) rehash();
    int idx = hash(key);
    for (auto& pair : buckets[idx])
        if (pair.first == key) { pair.second = value; return; }
    buckets[idx].push_back({key, value});
    count++;
}
C++ pseudocode — rehash():
void HashMap::rehash() {
    int newSize = bucketCount * 2;
    vector<list<pair<string,int>>> newBuckets(newSize);
    for (auto& bucket : buckets)
        for (auto& pair : bucket) {
            unsigned long h = 5381;
            for (char c : pair.first) h = ((h<<5)+h)^(unsigned char)c;
            newBuckets[h % newSize].push_back(pair);
        }
    buckets = move(newBuckets);
    bucketCount = newSize;
}
4.4 Stack — Packet History (LIFO)
Stack Quick Reference
Purpose: Store last N processed packets. LIFO: most recent shown first.
Implementation: Fixed-capacity circular array (avoids dynamic allocation overhead).
Capacity: Configurable, default MAX_HISTORY = 100.
Complexity: push O(1) | pop O(1) | peek O(1) | printTop(n) O(n)
C++ pseudocode — push() with circular overwrite:
void Stack::push(const Packet& p) {
    top = (top + 1) % capacity;
    items[top] = p;
    if (count < capacity) count++;
}
C++ pseudocode — printTop():
void Stack::printTop(int n) {
    int limit = min(n, count);
    int idx = top;
    for (int i = 0; i < limit; i++) {
        cout << items[idx].toString() << endl;
        idx = (idx - 1 + capacity) % capacity;
    }
}
4.5 Queue — Packet Buffer (FIFO)
Queue Quick Reference
Purpose: Buffer incoming packets when processing is temporarily slower than arrival.
Implementation: Fixed-capacity circular array.
Capacity: Configurable, default MAX_QUEUE = 1000.
Overflow policy: Drop packet silently, increment overflowCount.
Complexity: enqueue O(1) | dequeue O(1) | isEmpty O(1) | isFull O(1)
C++ pseudocode — enqueue():
bool Queue::enqueue(const Packet& p) {
    if (isFull()) { overflowCount++; return false; }
    rear = (rear + 1) % capacity;
    items[rear] = p;
    count++;
    return true;
}
C++ pseudocode — dequeue():
Packet Queue::dequeue() {
    if (isEmpty()) throw runtime_error("Queue underflow");
    Packet p = items[front];
    front = (front + 1) % capacity;
    count--;
    return p;
}

5. Algorithm Design
5.1 Core Algorithm: RuleEngine::evaluate()
This is the most critical algorithm in the system — every single packet runs through it. It must be both correct and fast.
Pseudocode:
Action RuleEngine::evaluate(const Packet& p) {
    // Step 1: Whitelist check — O(1)
    if (whitelist.contains(p.sourceIP)) {
        logEvent(p, ALLOW, WHITELIST_MATCH);
        return ALLOW;
    }
    // Step 2: Blacklist check — O(1)
    if (blacklist.contains(p.sourceIP)) {
        logEvent(p, BLOCK, BLACKLIST_MATCH);
        return BLOCK;
    }
    // Step 3: Rule match — O(log n)
    Rule* matched = ruleBST.findBestMatch(p);
    if (matched) {
        matched->hitCount++;
        logEvent(p, matched->action, matched->ruleID);
        return matched->action;
    }
    // Step 4: Default policy
    logEvent(p, defaultPolicy, DEFAULT_POLICY);
    return defaultPolicy;
}
Total worst-case: O(1) + O(1) + O(log n) = O(log n) per packet.
5.2 Packet Processing Pipeline: PacketProcessor::process()
Pseudocode:
void PacketProcessor::process(const Packet& p) {
    stats.totalProcessed++;
    // Stage 1: Validate packet
    if (!p.isValid()) {
        logger.logEvent(p, BLOCK, INVALID_PACKET);
        return;
    }
    // Stage 2: Enqueue for orderly processing
    if (!packetQueue.enqueue(p)) {
        logger.logEvent(p, BLOCK, QUEUE_OVERFLOW);
        stats.queueOverflows++;
        return;
    }
    // Stage 3: Dequeue and inspect
    Packet current = packetQueue.dequeue();
    // Stage 4: Threat detection
    ThreatType threat = monitor.inspect(current);
    if (threat != NONE) {
        engine.addToBlacklist(current.sourceIP);
        logger.logThreat(current.sourceIP, threat);
        if (threat == SYN_FLOOD)  stats.synFloodsDetected++;
        if (threat == PORT_SCAN)  stats.portScansDetected++;
    }
    // Stage 5: Evaluate rule
    Action decision = engine.evaluate(current);
    // Stage 6: Update counters
    if (decision == BLOCK) stats.totalBlocked++;
    else                   stats.totalAllowed++;
    // Stage 7: Push to history stack
    history.push(current);
}
5.3 Algorithm: Rule Matching — Rule::matches()
bool Rule::matches(const Packet& p) const {
    // Wildcard check for each field
    if (sourceIP != "*" && sourceIP != p.sourceIP) return false;
    if (destPort  != -1  && destPort  != p.destPort)  return false;
    if (protocol  != ANY && protocol  != p.protocol)  return false;
    if (direction != BOTH && direction != p.direction) return false;
    if (!enabled) return false;
    return true;
}
5.4 Algorithm: SYN Flood Detection — TrafficMonitor::inspect()
ThreatType TrafficMonitor::inspect(const Packet& p) {
    // Refresh window if expired
    time_t now = time(nullptr);
    if (difftime(now, windowStart) > windowSeconds) resetWindow();
    // SYN flood detection
    if (p.protocol == TCP && p.isSYN) {
        synCounters.increment(p.sourceIP);
        if (synCounters.get(p.sourceIP) > synThreshold)
            return SYN_FLOOD;
    }
    // Port scan detection
    portScanSets.increment(p.sourceIP + ":" + to_string(p.destPort));
    // Count distinct ports for this IP
    // (Simplified: track combined key, compare vs threshold * avg_port_repeat)
    int distinctPorts = countDistinctPorts(p.sourceIP);
    if (distinctPorts > portScanThreshold) return PORT_SCAN;
    return NONE;
}
5.5 Algorithm: IP Address Validation
5.6  Algorithm: HashMap::topN(int n)
Returns the N entries with the highest integer values. Used by show-stats to display top N blocked IPs or most-hit rules.
Pseudocode:
vector<pair<string,int>> HashMap::topN(int n):
  // Use a min-heap of size n to track top entries
  minHeap = priority_queue(size=n, comparator=by_value_ascending)
  for each bucket b in table[0..capacity-1]:
    for each Entry e in b.chain:
      if e.key is not empty:
        if minHeap.size() < n:
          minHeap.push({e.key, e.value})
        else if e.value > minHeap.top().value:
          minHeap.pop()
          minHeap.push({e.key, e.value})
  result = extract all entries from minHeap, sorted descending by value
  return result
Time Complexity: O(N log n) where N = total entries, n = requested top-N. Space: O(n). Degenerate case: if all N entries have same value, result is arbitrary n of them — acceptable for statistics display.
5.7  Algorithm: BST::findInOrderSuccessor(BSTNode* node)
Finds the in-order successor of a BST node — the node with the next-smallest priority value. Required by removeRule() when deleting a node that has two children.
Pseudocode:
BSTNode* BST::findInOrderSuccessor(BSTNode* node):
  // In-order successor is the leftmost node in the right subtree
  if node.right == nullptr: return nullptr  // no successor
  current = node.right
  while current.left != nullptr:
    current = current.left
  return current  // smallest in right subtree
// Used inside BST::remove(int priority):
// Case: node has two children
  successor = findInOrderSuccessor(node)
  node.priority = successor.priority    // copy successor data up
  node.rulePtr  = successor.rulePtr
  removeNode(node.right, successor.priority) // delete successor
Time Complexity: O(h) where h = tree height. O(log n) average, O(n) worst case (degenerate BST). Pre-condition: node has a right child. Post-condition: BST property maintained after deletion.
5.8  Algorithm: TrafficMonitor::countDistinctPorts(string sourceIP)
Returns the number of distinct destination ports that sourceIP has sent packets to in the current monitoring window. Used by port scan detection in TrafficMonitor::inspect().
Pseudocode:
int TrafficMonitor::countDistinctPorts(string sourceIP):
  // portScanSets: HashMap<string, unordered_set<int>>
  // Each entry maps sourceIP -> set of destPorts seen
  if portScanSets.contains(sourceIP) == false:
    return 0
  return portScanSets.get(sourceIP).size()
// Called from inspect() after each packet:
  portScanSets[p.sourceIP].insert(p.destPort)   // O(1) set insert
  distinctCount = countDistinctPorts(p.sourceIP) // O(1) set size
  if distinctCount > portScanThreshold:
    declarePORTSCAN(p.sourceIP)
Time Complexity: O(1) for insert and count (unordered_set). Space: O(U * D) where U = unique source IPs, D = avg distinct ports per IP. Degenerate case: single IP probing all 65,535 ports — set grows to 65,535 entries (< 1MB). Acceptable.
5.9  Algorithm: Stack::push() with Circular Overwrite
The history Stack has a fixed capacity. When full, the oldest entry is overwritten (circular buffer behaviour). This is an important design decision distinct from a standard stack.
Pseudocode:
void Stack::push(Packet p):
  if size < capacity:
    data[top] = p
    top = (top + 1) % capacity   // circular increment
    size++
  else:
    // Overwrite oldest entry (circular)
    data[top] = p                // top points to oldest slot
    top = (top + 1) % capacity
    // size stays at capacity (oldest entry lost)
Time Complexity: O(1) push always. Space: O(capacity) fixed. Design rationale: Fixed memory footprint is critical for a long-running network tool. A growing stack would eventually exhaust memory. Circular overwrite is the standard pattern for bounded history buffers (used in real tools like tcpdump ring buffers).
5.11  Algorithm: MongoLogger::logEvent()
Inserts one BSON document into the trafficlog collection for every packet decision. Designed for sub-5ms execution on a local MongoDB instance.
Pseudocode:
void MongoLogger::logEvent(const Packet p, Action a, int ruleID) {
  if (!connected) { FileLogger::logEvent(p, a, ruleID); return; }  // fallback
  try {
    auto doc = bsoncxx::builder::stream::document{}
      << "timestamp"     << p.timestamp
      << "packetID"      << p.packetID
      << "sourceIP"      << p.sourceIP
      << "destIP"        << p.destIP
      << "destPort"      << p.destPort
      << "protocol"      << protocolToStr(p.protocol)
      << "direction"     << directionToStr(p.direction)

}
Time complexity: O(1) single document insert into indexed collection. Network latency: less than 5ms on localhost.
5.12  Algorithm: MongoLogger::topNBlockedIPs()
Uses MongoDB aggregation pipeline to compute top-N blocked source IPs. Replaces HashMap::topN() computation for persistent log data.
Pseudocode:
vector MongoLogger::topNBlockedIPs(int n) {
  mongocxx::pipeline p;
  p.match(doc{} << "action" << "BLOCK");
  p.group(doc{} << "_id" << "$sourceIP" << "count" << doc{} << "$sum" << 1);
  p.sort(doc{} << "count" << -1);
  p.limit(n);
  auto cursor = trafficlogCol.aggregate(p);
  vector results;
  for (auto doc : cursor)
    results.push_back({doc["_id"].get_string(), doc["count"].get_int32()});
  return results;
}
Equivalent SQL: SELECT sourceIP, COUNT(*) FROM trafficlog WHERE action=BLOCK GROUP BY sourceIP ORDER BY COUNT(*) DESC LIMIT n;
5.10  Degenerate Case Analysis
The following table documents worst-case scenarios for each DSA structure and the system-level mitigation.
Structure
Degenerate Case
Complexity & Mitigation
BST
All rules inserted in ascending priority order — tree becomes a right-skewed linked list

HashMap (blacklist)

bool AdminCLI::validateIP(const string& ip) {
    istringstream ss(ip);
    string token;
    int parts = 0;
    while (getline(ss, token, '.')) {
        parts++;
        if (parts > 4) { cerr << "Error: too many octets.\n"; return false; }
        int val;
        try { val = stoi(token); }
        catch (...) { cerr << "Error: non-numeric octet '" << token << "'\n"; return false; }
        if (val < 0 || val > 255) {
            cerr << "Error: octet " << val << " exceeds 0-255 range.\n";
            return false;
        }
    }
    if (parts != 4) { cerr << "Error: expected 4 octets, got " << parts << "\n"; return false; }
    return true;
}

8. Error Handling Design
8.1  Exception Class Hierarchy
All exceptions inherit from FirewallException, which inherits from std::exception. This allows catch-all handlers at the top level while enabling specific handling at lower levels.
Class hierarchy (defined in errors.h):
std::exception
  └── FirewallException          // base: message + component
        ├── ValidationException  // IP/port/rule format errors
        ├── DataStructureException // BST, HashMap, Stack, Queue errors
        │     ├── OverflowException   // Stack/Queue full
        │     └── UnderflowException  // pop/dequeue on empty
        └── IOException          // file read/write failures
              ├── FileNotFoundException
              └── ParseException       // malformed rules.conf line
Usage pattern in AdminCLI::dispatch():
try {
  engine.addRule(r);
} catch (const ValidationException& e) {
  cerr << "[VALIDATION] " << e.what() << endl;
} catch (const DataStructureException& e) {

13.5  Risk Assessment and Mitigation
The following risks are specific to the design decisions documented in this SDS. Each is assessed for probability and has a defined mitigation.
Risk ID | Description
Probability | Mitigation
RISK-01: BST O(n) degeneration — rules inserted in sorted priority order
Probability: Medium. Mitigation: Insert rules in random/varied order during tests. Note AVL self-balancing as a planned enhancement in viva. Current worst-case O(n) is acceptable for < 1,000 rules.
RISK-02: HashMap collision storm — many IPs share the same djb2 bucket
Probability: Low. Mitigation: djb2 is proven for string distribution. Rehash triggered at load factor 0.75. Monitor loadFactor() via show-stats in production.
RISK-03: Stack circular overwrite — history lost in high-traffic burst
Probability: Medium. Mitigation: MAX_HISTORY is a compile-time constant — increase before high-volume test. Document the bounded-history trade-off in system documentation.
RISK-04: rules.conf parse failure corrupts loaded rule set
Probability: Low. Mitigation: Parser skips malformed lines with stderr warning. Validate loaded rule count equals saved count. Atomic write (write to temp file then rename) is future enhancement.
RISK-05: SYN threshold false positives on legitimate high-SYN traffic
Probability: Medium. Mitigation: Default threshold=100 is conservative. set-threshold command allows runtime tuning without restart. Document recommended values.
RISK-06: Memory leak in dual-structure node deletion
Probability: Low. Mitigation: BSTNode stores rulePtr as non-owning pointer. Only LinkedList::remove() frees the Rule/RuleNode. Unit test UT-06 verifies size after 50 removals.
RISK-07: FileLogger ofstream left open on abnormal exit
Probability: Low. Mitigation: RAII destructor closes stream. Signal handler (SIGINT) calls logger.close() before exit to flush pending writes.
Appendix B — Quick Viva Reference
If asked in the DSA viva: Why did you use a BST for rule storage? Answer: Rules need to be evaluated in priority order for every packet. A BST gives O(log n) ordered access, compared to O(n) for a LinkedList scan. The BST's in-order traversal naturally gives rules from highest to lowest priority.
If asked: Why HashMap for the blacklist? Answer: The blacklist is checked on every single packet before any rule evaluation. It needs O(1) average-case lookup to avoid being the performance bottleneck.
If asked: Why Stack for history? Answer: Administrators ask to see the LAST N events — this is a LIFO access pattern. A Stack is the correct structure. Circular array implementation avoids memory allocation overhead.
If asked: Why Queue for packet buffering? Answer: Packets must be processed in arrival order — FIFO. When processing is slower than arrival (burst), a Queue buffers them without losing order.
If asked: Why LinkedList at all, given BST? Answer: The LinkedList is the canonical ordered list for display and serialisation. When saving rules to file, you need sequential access. The BST is optimised for search, not sequential output.
