1.  Introduction
This Software Requirements Specification (SRS) defines all functional and non-functional requirements for the C++ Packet Filtering Firewall — a command-line application that inspects simulated network packets, enforces configurable security rules using five core data structures (LinkedList, BST, HashMap, Stack, Queue), detects anomalous traffic patterns, and logs all decisions to persistent files. This document simultaneously satisfies CLO-2 of CSC-291 (Software Engineering Concepts) and supports the DSA, DBMS, and Information Security subjects.
1.1  Purpose
The goal of this SRS is to clearly identify all functional and non-functional requirements for the C++ Packet Filtering Firewall. It provides a precise specification that guides implementation, ensures all stakeholders share a common understanding of the project's objectives, and serves as the acceptance baseline for testing. Stakeholders include: the implementing developer, the course supervisor, testers, and documentation writers.
1.2  Scope
The firewall accepts simulated network packets (C++ structs), evaluates them against a configurable rule set, applies ALLOW or BLOCK decisions, detects SYN flood and port scan attacks, maintains real-time packet history and burst buffering, and logs every decision to a CSV file. Rules and packet logs persist in MongoDB collections (rules, trafficlog, blacklist, whitelist, threats). The flat-file backup (rules.conf / firewall.log) is retained as a secondary fallback. The system does not implement live packet capture, stateful TCP tracking, deep packet inspection, IPv6, or a GUI (planned future increment).

2.  Overall Description
This section provides a high-level overview of the C++ Packet Filtering Firewall, covering product context, user classes, operating environment, and design constraints.
2.1  Product Perspective
The C++ Packet Filtering Firewall is a standalone academic system simulating a stateless network firewall. It is the primary DSA major project, extended to integrate SE, DBMS, and InfoSec concepts. Unlike enterprise systems (Palo Alto NGFW, FortiGate), it operates on simulated packets rather than live network interfaces. It is architecturally designed using a Pipes-and-Filters + Layered hybrid with MVC separation so the CLI can be replaced by a GUI without touching the engine.
2.2  User Classes and Characteristics
Network Administrator: Primary CLI user. Configures rules, manages blacklists/whitelists, monitors statistics, and queries logs. Requires basic networking knowledge (IP addresses, ports, TCP/UDP).
Security Analyst: Queries the traffic log to investigate suspicious activity. Uses query commands to filter by IP, time, or action.
Developer / DSA Evaluator: Reviews source code and internal data structure implementations for correctness during the project viva.
Academic Supervisor / Examiner: Assesses SRS and documentation for completeness and alignment with CLO-2.
2.3  Operating Environment
Hardware: Any modern desktop/laptop with ≥2 GB RAM and dual-core CPU. No dedicated network hardware required.
Operating System: Linux (Ubuntu 20.04+, GCC 9+) and Windows (MinGW GCC 9+ or MSVC). Only C++17 standard library used — no platform-specific APIs in the core engine.
Persistence: Two plain-text files — firewall.log (CSV, append-only) and rules.conf (human-readable config), both in the working directory.

2.4  Design and Implementation Constraints
CON-1: Implemented entirely in C++17 standard library only. No external libraries for core data structures.
CON-2: All five data structures (LinkedList, BST, HashMap, Stack, Queue) are custom implementations — STL containers may not replace them.
CON-3: Operates on simulated packets (C++ structs). No live packet capture via libpcap or raw sockets.
CON-4: Primary persistence uses MongoDB (mongocxx driver, C++17). Flat-file backup (rules.conf / firewall.log) retained as fallback when MongoDB is unavailable.
CON-5: Single-threaded. Concurrent access is not supported.
CON-6: IPv4 only. All IPs are dotted-decimal strings.

Module 1:  Packet Filtering Engine
4.2.1.1  UC-1: Filter Packet by Source IP
Use Case ID
UC-1
Use Case Name
Filter Packet by Source IP
Actors
Primary: Network (source of packets)  |  Secondary: System (PacketProcessor, RuleEngine, HashMap)
Description
The system inspects each incoming packet's source IP address and checks it against the whitelist and blacklist HashMaps in O(1) average time, then against the BST rule set in O(log n).
Trigger
A new Packet object arrives at PacketProcessor::process().
Level
High
Preconditions
PRE-1: Firewall engine is running.  PRE-2: At least the blacklist HashMap is initialised.
Postconditions
POST-1: Packet is ALLOWED or BLOCKED.  POST-2: Log entry appended to firewall.log.  POST-3: Packet pushed onto history Stack.
Normal Flow
1. Packet arrives at PacketProcessor.  2. whitelist.contains(sourceIP) → if true, ALLOW immediately.  3. blacklist.contains(sourceIP) → if true, BLOCK immediately.  4. BST::findBestMatch() traversed in-order.  5. First matching rule action applied.  6. Default policy applied if no rule matches.  7. Event logged, packet pushed to Stack.
Alternative Flows
AF-1: Rule has wildcard sourceIP (*) → matches all source IPs.
Exceptions
E-1: Packet::isValid() returns false (malformed IP) → packet dropped with INVALID log entry.
Business Rules
BR-1: Whitelist checked before blacklist.  BR-2: Blacklist checked before BST traversal.
Assumptions
90% of packets evaluated via O(1) HashMap check before reaching BST.
4.2.1.2  UC-2: Filter Packet by Port and Protocol
Use Case ID
UC-2
Use Case Name
Filter Packet by Destination Port and Protocol
Actors
Primary: Network  |  Secondary: System (RuleEngine, BST)
Description
The system matches incoming packets against rules specifying destination port and/or protocol type. The BST is traversed in priority order; the first matching rule determines the action.
Trigger
PacketProcessor calls RuleEngine::evaluate() after whitelist/blacklist checks pass.
Level
High
Preconditions
PRE-1: Rules exist in the BST with non-wildcard port or protocol fields.
Postconditions
POST-1: Rule with highest priority matching port and protocol is applied.  POST-2: Rule hitCount incremented.
Normal Flow
1. BST::findBestMatch() begins in-order traversal.  2. Rule::matches() checks destPort and protocol for each node.  3. First matching rule returned.  4. Rule action (ALLOW/BLOCK) applied.  5. hitCount incremented, decision logged.
Alternative Flows
AF-1: destPort = -1 (wildcard) → matches any port.  AF-2: protocol = ANY → matches any protocol.
Exceptions
E-1: No rule matches + DEFAULT_DENY → BLOCK applied.
Business Rules
BR-1: First-match wins (lowest priority number = highest precedence).  BR-2: destPort -1 is stored as integer -1 meaning ANY.
Assumptions
BST is balanced on average; O(log n) evaluation for 1000+ rules.

4.2.1.4  UC-4: Inbound and Outbound Direction Filtering
Use Case ID
UC-4
Use Case Name
Apply Direction-Based Filtering (INBOUND / OUTBOUND / BOTH)
Actors
Primary: Network  |  Secondary: System (RuleEngine)
Description
Rules carry a direction field (INBOUND / OUTBOUND / BOTH). During BST evaluation Rule::matches() rejects rules whose direction does not match the packet's direction, ensuring inbound-only rules do not affect outbound traffic.
Trigger
PacketProcessor::process() called with Packet whose direction field is set.
Level
Medium
Preconditions
PRE-1: Packet has direction field set to INBOUND or OUTBOUND.  PRE-2: At least one directional rule exists in BST.
Postconditions
POST-1: Only rules matching the packet's direction are considered during evaluation.
Normal Flow
1. Packet arrives with direction = INBOUND.  2. BST traversal calls Rule::matches() for each node.  3. Rules with direction = OUTBOUND return false.  4. First INBOUND or BOTH rule matching other fields is applied.
Alternative Flows
AF-1: Rule direction = BOTH → matches any direction.
Exceptions
E-1: Packet direction uninitialised → treated as INBOUND by default.
Business Rules
BR-1: BLOCK OUT TCP *:25 blocks outbound SMTP without affecting inbound.  BR-2: BOTH is the default direction for new rules.
Assumptions
70% of admin-defined rules specify direction explicitly.

5.  Non-Functional Requirements
This section specifies all quality attributes and constraints that govern how the system performs, beyond what it does. Requirements are grouped by quality category.
5.1  Reliability
5.1.1  Measurable Acceptance Criteria
Each NFR is supplemented with a concrete, testable Acceptance Criterion (AC).
NFR ID
Acceptance Criterion (AC)
REL-1
1,000 crafted invalid packets (bad IP, overflow port, null fields) — 100% rejected; zero exceptions, zero crashes.
REL-2
Enqueue 1,001 packets into cap-1,000 Queue. overflowCount == 1. System continues processing normally.
REL-3
Mock disk-full condition (ofstream fail). Error printed to stderr. System continues processing. Does not exit.
REL-4
Delete rules.conf before launch. System starts with 0 rules, DEFAULT_DENY active. Warning message within 1 second.
REL-6
addRule() called 1,000 times. After each call: LinkedList.size == BST.nodeCount. No structural divergence.
REL-8
Push 10,000 packets into Stack cap=100. No exception. Most-recent 100 always accessible via printTop().
REL-9
Insert 100,000 IPs into HashMap. Load factor <= 0.75 before each rehash. All entries retrievable after.
USE-3
Test user reads help add-rule, types valid command, verifies with send-packet, reads show-history — all in under 5 minutes.
PER-1
10,000-packet throughput test with 1,000 rules loaded. Median evaluate() < 1ms. 99th percentile < 5ms.
PER-2
HashMap::contains() called 100,000 times on 100,000-entry map. All individual calls complete in < 0.1ms.
PER-3
BST with 1,000 rules. findBestMatch() visits at most ceil(log2(1000)) = 10 nodes per call.
SEC-5
Launch without rules.conf. Default policy must be DENY. Unmatched packet must log action=BLOCK.
SCAL-1
Load 10,000 rules from file. list-rules completes < 10 seconds. send-packet evaluation < 15ms.
REL-1: The system shall handle all invalid packet inputs (malformed IPs, out-of-range ports, null pointers) without crashing. All invalid packets shall be dropped gracefully with an appropriate INVALID log entry.
REL-2: Queue overflow shall be handled non-fatally. The overflow packet is dropped, overflowCount incremented, and processing continues without interruption.
REL-3: Log file write failures (e.g., disk full, permission denied) shall not crash the application. The system shall print an error to stderr and continue operation. Log entries lost due to disk failure are silently counted.
REL-4: If rules.conf is missing at startup, the system shall start with an empty rule set and DEFAULT_DENY policy, displaying a clear Warning: rules.conf not found message.
REL-5: Malformed lines in rules.conf shall be skipped with a per-line Warning: skipping malformed line N message. All valid lines shall be successfully loaded.
REL-6: Data structure consistency shall be maintained at all times: every addRule() operation succeeds in both the LinkedList and BST atomically, or fails in both.
REL-7: The system shall not lose any in-memory rule state during normal CLI operation. Only an application crash or explicit flush-rules command shall clear the rule set.
REL-8: The Stack shall never throw on push, even when full. Circular overwrite ensures push() is always O(1) and exception-free.
REL-9: The HashMap shall never fail to insert a new entry due to capacity issues. Automatic rehashing when load factor exceeds 0.75 ensures unlimited insertions.
REL-10: The system shall correctly persist and restore all rule set state across save/load cycles. A saved rule set shall be bit-for-bit equivalent to a loaded rule set in terms of functional behaviour.
5.2  Usability
USE-1: Every CLI command shall provide a descriptive error message when input validation fails. The message shall specify exactly which field is invalid and why (e.g., 'Error: octet 999 exceeds the 0–255 range').
USE-2: The help command shall display a usage summary for every command. help <command> shall show full syntax, parameter descriptions, and a concrete example.
USE-3: A new administrator shall be able to add their first firewall rule, verify it with send-packet, and view the result in show-history within 5 minutes of starting the application, using only the help command for guidance.
USE-4: All IP addresses shall be displayed in standard dotted-decimal notation. Port numbers shall appear as decimal integers. No raw hex, binary, or enum integer values shall appear in CLI output.
USE-5: The list-rules command shall display rules in a clearly formatted table with consistent column widths, making every field immediately legible.
USE-6: The show-stats command shall display statistics in a two-column label:value format with all values right-aligned.
USE-7: Unrecognised commands shall print 'Unknown command. Type help for a list of commands.' without a stack trace or crash.
USE-8: Empty input (pressing Enter with no text) shall cause the prompt to reappear silently without an error message.
USE-9: The exit command shall display 'Saving rules... Goodbye.' before terminating, confirming auto-save has occurred.
USE-10: Status confirmations shall use consistent phrasing: 'Rule #N added.', 'IP X.X.X.X blocked.', 'Saved N rules to rules.conf.'
5.3  Performance
PER-1: Single packet evaluation latency (from PacketProcessor::process() entry to log write) shall be under 1 millisecond for rule sets of up to 1,000 rules on any modern desktop CPU.
PER-2: IP blacklist/whitelist lookup (HashMap::contains) shall complete in O(1) average time. Under load with 100,000 IPs, single lookup shall complete in under 0.1 milliseconds.
PER-3: BST-based rule evaluation shall achieve O(log n) average time. For n = 1,000 rules, at most 10 comparisons per packet.
PER-4: Application startup including loading 1,000 rules from rules.conf and rebuilding all data structures shall complete in under 2 seconds.
PER-5: The show-stats command shall return all statistics in under 100 milliseconds regardless of log file size (all statistics are in-memory counters).
PER-6: Log query commands (query --ip, query --action, query --last) shall complete in under 3 seconds for log files containing up to 100,000 entries on a standard SSD.
PER-7: HashMap rehashing (triggered at load factor > 0.75) shall complete transparently within an acceptable time. For 100,000 entries, rehash shall complete in under 500 milliseconds.
PER-8: The Stack push() and pop() operations shall complete in O(1) time. The circular array implementation avoids any dynamic memory allocation at runtime.
PER-9: The Queue enqueue() and dequeue() operations shall complete in O(1) time using circular array indexing.
PER-10: Threat detection overhead per packet (TrafficMonitor::inspect()) shall not exceed 1 millisecond for any input, as all operations inside inspect() are O(1) HashMap operations.
5.4  Security
SEC-1: All CLI input strings shall be sanitised before processing. No shell injection, buffer overflow via long strings, or format string vulnerabilities shall be exploitable through CLI commands.
SEC-2: The log file (firewall.log) shall be append-only. No CLI command shall expose a delete or truncate operation, preserving audit trail integrity (CIA Integrity pillar).
SEC-3: A maliciously crafted Packet struct (e.g., sourceIP containing null bytes, port = INT_MAX) shall be rejected by Packet::isValid() and dropped without affecting system state.
SEC-4: The rules.conf file parser shall reject any line not conforming to the defined format. It shall not execute embedded data as code.
SEC-5: The system shall default to DEFAULT_DENY policy at startup, ensuring that the most secure configuration is active unless the administrator explicitly changes it (principle of least privilege).
SEC-6: The system shall implement the CIA Triad as follows — Confidentiality: blocking unauthorised access to services via port rules; Integrity: SYN flood detection prevents connection table exhaustion; Availability: whitelist and Queue buffering preserve legitimate traffic during attacks.
SEC-7: Auto-blocked IPs (from threat detection) shall only be removable via an explicit remove-block <ip> command, preventing accidental unblocking during attacks.
SEC-8: The whitelist shall never be writable by the threat detection auto-block mechanism. Auto-block writes only to the blacklist, never the whitelist.
5.5  Maintainability
MAIN-1: New rule filtering criteria (e.g., MAC address, VLAN ID) shall be addable by modifying only the Rule class and Rule::matches() method, without touching PacketProcessor, FileLogger, or AdminCLI.
MAIN-2: The logging backend shall be replaceable (file logger → database logger) by implementing a common Logger interface, without modifying PacketProcessor.
MAIN-3: Every public method shall have a Doxygen-style comment block explaining its purpose, parameters, return value, and any exceptions it may throw.
MAIN-4: The cyclomatic complexity of any single function shall not exceed 10. Functions exceeding this threshold shall be refactored into smaller sub-functions.
MAIN-5: The system shall use constructor injection for all inter-module dependencies (PacketProcessor receives RuleEngine, TrafficMonitor, FileLogger, Stack, Queue by reference). No global variables.
5.6  Scalability
SCAL-1: The system shall support up to 10,000 active firewall rules stored in the BST (approximately 13 tree levels). Performance at this scale: O(log 10,000) ≈ 14 comparisons per lookup.
SCAL-2: The blacklist and whitelist HashMaps shall support up to 100,000 IP entries each with load factor ≤ 0.75 before automatic rehashing doubles the bucket count.
SCAL-3: The log file shall support append-only growth up to 1 GB without write performance degradation (sequential appends are independent of file size).
SCAL-4: The LinkedList shall support up to 10,000 rules without structural issues. list-rules execution time scales linearly O(n) with rule count.
SCAL-5: The Stack and Queue shall support configurable capacities at startup. Default capacities (Stack: 100, Queue: 1,000) are sufficient for typical academic demonstration loads.
5.7  Portability
PORT-1: The system shall compile and run without modification on Linux (Ubuntu 20.04+, GCC 9+) and Windows (MinGW GCC 9+ or MSVC 2019+).
PORT-2: The system shall use only C++17 standard library headers. No POSIX-only functions (e.g., unistd.h), Windows-only APIs, or platform-specific compiler extensions shall be used in the core engine.
PORT-3: File paths shall use relative paths only (./firewall.log, ./rules.conf). No hardcoded absolute paths shall appear in the code.
PORT-4: The system shall use std::time() and std::difftime() for all time operations (C standard library, cross-platform).

7.  Conclusion
This Software Requirements Specification defines a complete and academically rigorous set of functional and non-functional requirements for the C++ Packet Filtering Firewall — a system that integrates Data Structures and Algorithms, Software Engineering, Database Management Systems, and Information Security into a single cohesive project.
The document specifies 180 Functional Requirements across six modules (30 per module), 36 complete tabular Use Case specifications, six Use Case Diagram specifications, seven NFR categories covering reliability, usability, performance, security, maintainability, scalability, and portability, and four external interface requirement sections.
All five required DSA data structures are formally specified with their roles, operations, time complexities, and C++ pseudocode in the companion SDS document. The LinkedList provides O(n) ordered traversal for rule listing and serialisation. The BST provides O(log n) priority-keyed rule lookup for every packet decision. The HashMap provides O(1) average IP blacklist, whitelist, and threat detection counter operations. The Stack provides O(1) LIFO packet history. The Queue provides O(1) FIFO packet buffering.
